**File 1: motif-system-architect.md**

Read the current file. Add a new "## Brownfield Mode" section AFTER the "## Domain Expertise" section and BEFORE "## Output Format Expectations". This section teaches the agent how to handle each token strategy.

Add:
```markdown
## Brownfield Mode

When TOKEN-INVENTORY.md is provided in your context with a strategy directive:

### Strategy: Adopt Existing
- Read all tokens from TOKEN-INVENTORY.md
- Map them to Motif token names using the Motif Equivalent column
- Generate ONLY tokens that have no equivalent in the inventory
- Preserve user token NAMES — if user has --brand-blue, keep it AND add an alias: `--color-primary-500: var(--brand-blue);`
- Comment preserved tokens: `/* Preserved from project: [source file] */`
- Note in tokens.css header: `/* Strategy: ADOPT — preserving all existing tokens, generating gaps only */`

### Strategy: Merge (Default)
- Read all tokens from TOKEN-INVENTORY.md
- For each Motif category, check coverage from the inventory
- Generate missing tokens to complete the system
- Preserve existing VALUES for matched tokens but use Motif naming
- Use existing token values to DERIVE compatible new tokens (e.g., if user's primary is blue, generate a blue-based scale, not random teal)
- Detect user's spacing base (4px vs 8px vs other) and generate gap-fill tokens on the SAME base
- Comment each token: `/* Preserved from project */` or `/* Generated by Motif */`
- Note in tokens.css header: `/* Strategy: MERGE — preserving existing + filling gaps */`

### Strategy: Fresh
- Ignore TOKEN-INVENTORY.md entirely
- Generate as if greenfield — use all standard decision algorithms
- Note in tokens.css header: `/* Strategy: FRESH — existing project tokens not incorporated */`

### When COMPONENT-GAP.md is provided:
- For components listed as "Existing": generate a REFERENCE spec only — name, file path, note "existing in project — no generation needed". Use minimal tokens (~50 per component).
- For components listed as "Missing": generate FULL spec (variants, states, accessibility) as normal.
- For components listed as "Partial Match": generate FULL spec with a note: `<!-- Partial match: [found-name] at [file-path]. May need enhancement. -->`
- This reduces COMPONENT-SPECS.md size for brownfield projects with many existing components.

### Anti-patterns for Brownfield:
- NEVER overwrite a user's existing token value in merge mode — the entire point of merge is preservation
- NEVER use user's font choices to override Motif's font ban list in FRESH mode — fresh means fresh
- NEVER generate full component specs for components marked "existing" — this wastes token budget
- NEVER ignore the user's spacing base in merge mode — if they use 8px grid, fill gaps on 8px grid
```
