<task type="auto">
  <name>Task 1: Build token-extractor.js script</name>
  <files>scripts/token-extractor.js</files>
  <action>
Create `scripts/token-extractor.js` — a zero-dependency Node.js script following the exact same patterns as `scripts/project-scanner.js` (CLI entry, fs/path only, SKIP_DIRS, walkProject helper, safeReadFile helper).

The script accepts `[project-root]` and `[--output-dir <dir>]` arguments (same as project-scanner.js).

**Three extraction pipelines:**

1. **CSS Custom Property Extraction:**
   - Walk project for .css and .scss files (reuse SKIP_DIRS from project-scanner.js pattern)
   - Extract `--name: value;` declarations using regex: `/--([a-zA-Z0-9_-]+)\s*:\s*([^;]+);/g`
   - Track source file for provenance
   - Detect if token is in global scope (:root, html, :host) vs component-scoped
   - Flag inconsistent tokens (same name, different values across files)

2. **Tailwind Config Extraction:**
   - Check for tailwind.config.js, .ts, .mjs, .cjs
   - Extract theme.extend and theme override sections via regex (NOT require() — see research pitfall 3)
   - Parse simple key-value pairs from object literals
   - Handle both v3 (JS config) and v4 (@theme blocks in CSS — already handled by CSS extraction since v4 outputs standard custom properties)
   - If config uses dynamic values (require(), spread, functions), flag confidence as LOW

3. **JS Theme File Extraction (best-effort):**
   - Only if PROJECT-SCAN.md indicates styled-components or Emotion CSS approach
   - Look for common theme file patterns: src/theme.ts, src/styles/theme.js, styled.d.ts
   - Extract key-value pairs from exported objects via regex
   - Flag as LOW confidence

**Token categorization:**
Map discovered tokens to Motif categories using the MOTIF_TOKEN_CATEGORIES pattern from the research (colors, typography, spacing, radii, shadows, transitions). Use both name-matching (patterns like /color/, /bg/, /primary/) AND value-based heuristics (hex values in blue range = possible primary). See research Pattern 4 for the full category mapping.

**Coverage calculation:**
For each Motif category, calculate how many required tokens are matched vs missing. Required token lists come from the research's MOTIF_TOKEN_CATEGORIES.required arrays.

**Output: TOKEN-INVENTORY.md**
Write to `{outputDir}/.planning/design/TOKEN-INVENTORY.md` following the exact format from the research "TOKEN-INVENTORY.md Output Format" section. Include:
- Summary (counts, coverage percentage)
- Per-category tables (Token | Value | Source | Motif Equivalent)
- Coverage and Missing lists per category
- Tailwind theme customizations table (if detected)
- Token budget: target ~1000 tokens, cap at 1500

**Edge cases:**
- Zero tokens found: exit cleanly with message "No design tokens found", do NOT create TOKEN-INVENTORY.md
- CSS-in-JS detected but no theme files found: report limitation message
- Tailwind config too complex to parse: report "detected but could not fully parse" with LOW confidence flag

**CLI output:** Print summary to stdout (N CSS tokens, N Tailwind tokens, coverage percentage) for the scan workflow to read.
  </action>
  <verify>
Run `node scripts/token-extractor.js .` on the current project. It should either produce a TOKEN-INVENTORY.md or cleanly report no tokens found. No crashes, no unhandled errors. Check that SKIP_DIRS are respected (no node_modules scanning).
  </verify>
  <done>
token-extractor.js exists, runs without errors on any project directory, produces correctly formatted TOKEN-INVENTORY.md when tokens exist, exits cleanly when no tokens found.
  </done>
