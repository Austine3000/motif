# Phase 12: Enforcement and Validation - Research

**Researched:** 2026-03-04
**Domain:** Icon enforcement in design reviewer agent + accessibility hook extension + end-to-end pipeline validation
**Confidence:** HIGH

## Summary

Phase 12 closes the v1.1 Icon Library Integration milestone by adding enforcement to the two validation layers that already exist in the Motif pipeline: the design reviewer agent (Lens 3 + Lens 4 checks) and the aria-check PostToolUse hook (HOOK-03). Phases 9-11 built the foundation (icon-libraries.md), curated per-vertical vocabularies (22-23 icons each), and wired the system architect and screen composer to produce and consume ICON-CATALOG.md. What is missing is the validation loop -- the reviewer does not currently check icon names or vertical appropriateness, the aria-check hook does not detect icon elements, and no end-to-end test exists to prove the full pipeline works.

The domain is well-scoped: two existing files need targeted extensions (motif-design-reviewer.md, motif-aria-check.js), the review.md workflow needs an icon-related lens addition, and a single end-to-end validation run confirms everything works together. The primary risk is false positives in the aria-check hook -- specifically, text buttons that contain adjacent icons should NOT be flagged as "icon-only buttons missing aria-label" because the visible text already provides the accessible name.

**Primary recommendation:** Add icon consistency checks to the reviewer agent (catalog validation + vertical appropriateness), extend the aria-check hook with icon element detection patterns for all 4 library syntaxes, and validate with one end-to-end pipeline run on a vertical.

### Pattern 1: Icon Consistency Checks in Reviewer (IENF-01)

**What:** The reviewer agent extracts all icon references from the screen source code, cross-references them against ICON-CATALOG.md, and flags any icon names not in the catalog. It also checks that icon choices are appropriate for the project's vertical.

**When to use:** During Lens 3 (Design System Compliance) and Lens 4 (Vertical UX Compliance).

**Where the checks go:**

1. **Lens 3 addition -- Icon Catalog Compliance:** Grep the source code for icon elements matching the 4 library patterns. Extract icon names. Compare each against ICON-CATALOG.md. Flag any name not in the catalog. Also check for: hardcoded icon sizes (px instead of --icon-* tokens), missing CDN link, bracket placeholders.

2. **Lens 4 addition -- Icon Vertical Appropriateness:** Check whether the icon semantic roles used match the project's vertical. A health app using `ph-credit-card` or `ph-chart-line-up` (fintech icons) should be flagged. A fintech app using `medication` or `stethoscope` (health icons) should be flagged.

**Detection regex patterns for extracting icon names from source code:**
```
Phosphor:    class="ph[-\w]* ph-([\w-]+)"       → extracts icon name
Lucide:      data-lucide="([\w-]+)"              → extracts icon name
Material:    class="material-symbols-\w+">([\w_]+)</span>  → extracts icon name
Tabler:      class="ti ti-([\w-]+)"              → extracts icon name
```

**Vertical appropriateness check approach:**
- Read the vertical from PROJECT.md (already in always_load for reviewer)
- Read ICON-CATALOG.md to know which icons are in the project's vocabulary
- Any icon name in the source code NOT in the catalog is either: (a) hallucinated, or (b) from the wrong vertical
- The reviewer flags it as a Major issue with an exact fix: "Replace `{wrong-icon}` with `{closest-match-from-catalog}`"

### Pattern 2: Icon Accessibility Checks in aria-check Hook (IENF-02)

**What:** Extend the existing aria-check hook to detect icon elements and enforce:
- Icon-only buttons MUST have `aria-label`
- Decorative icons (icons with adjacent text) MUST have `aria-hidden="true"`

**How to detect icon elements across all 4 libraries:**
```javascript
// Phosphor: <i class="ph ph-{name}"></i> or <i class="ph-{weight} ph-{name}"></i>
const phosphorPattern = /<i\b[^>]*class="[^"]*\bph[-\s][^"]*"[^>]*>/gi;

// Lucide: <i data-lucide="{name}"></i>
const lucidePattern = /<i\b[^>]*data-lucide="[^"]*"[^>]*>/gi;

// Material Symbols: <span class="material-symbols-{style}">{name}</span>
const materialPattern = /<span\b[^>]*class="[^"]*material-symbols-[^"]*"[^>]*>/gi;

// Tabler: <i class="ti ti-{name}"></i>
const tablerPattern = /<i\b[^>]*class="[^"]*\bti\b[^"]*"[^>]*>/gi;
```

**Critical false positive prevention:**
The phase description explicitly warns: "aria-check hook icon detection must avoid false positives on text buttons with adjacent icons." This means:

```html
<!-- This is NOT an icon-only button -- has visible text "Settings" -->
<button><i class="ph ph-gear"></i> Settings</button>
<!-- DO NOT flag this for missing aria-label -->

<!-- THIS is an icon-only button -- no visible text -->
<button><i class="ph ph-gear"></i></button>
<!-- Flag: icon-only button missing aria-label -->
```

**Detection strategy for icon-only buttons:**
1. Find all `<button>` elements containing icon elements
2. Strip the icon element from the button content
3. Check if any visible text remains after stripping
4. If NO visible text remains -> flag as icon-only button needing `aria-label`
5. If visible text exists -> the icon is decorative, check for `aria-hidden="true"` on the icon

**Detection strategy for decorative icons (standalone, not in buttons):**
1. Find all icon elements NOT inside `<button>` elements
2. If the icon is adjacent to text content (e.g., inside a `<span>`, `<a>`, `<li>` with other text)
3. The icon should have `aria-hidden="true"`
4. Standalone informative icons (icon without any adjacent text, not in a button) should have `aria-label` or `role="img"` + title

### Pattern 3: End-to-End Validation

**What:** Run the full pipeline (/motif:system -> /motif:compose -> /motif:review) on at least one vertical and verify zero icon-related review findings.

**This is a manual/human test** -- it requires actual agent invocations. The planner should define the test procedure, expected outcomes, and verification criteria. The verification script can check the resulting REVIEW.md for icon-related findings.

### Anti-Patterns to Avoid

- **Over-broad icon detection in the hook:** Do NOT match every `<i>` or `<span>` tag. Only match the 4 specific library patterns. A generic `<i>` tag for italics or a `<span>` with no icon class must not trigger.
- **Flagging icon buttons inside `<a>` tags with href text:** An `<a href="..."><i class="ph ph-link"></i> Link text</a>` has visible text -- the icon is decorative. Only flag `<a>` elements where the icon is the ONLY content.
- **Making the aria-check hook project-aware:** The hook should NOT read ICON-CATALOG.md or PROJECT.md. It operates on raw HTML patterns only. Catalog validation is the reviewer's job (batch check at review time), not the hook's job (real-time check at write time).
- **Adding icon checks to the token-check hook:** Icon name validation is NOT a token check. Tokens are CSS custom properties. Keep hook responsibilities cleanly separated.

## Common Pitfalls

### Pitfall 1: False Positives on Text Buttons with Adjacent Icons
**What goes wrong:** The hook flags `<button><i class="ph ph-x"></i> Close</button>` as missing `aria-label` because it detects an icon inside a button.
**Why it happens:** Naive detection that checks "is there an icon in a button?" without checking "is there also visible text?"
**How to avoid:** After finding an icon inside a button, strip the icon element from the button's inner HTML. If only whitespace remains, it is icon-only. If text remains, the icon is decorative.
**Warning signs:** During testing, common UI patterns like "Save" buttons with a floppy disk icon would produce false alerts.

### Pitfall 2: Material Symbols Syntax Confusion
**What goes wrong:** Material Symbols uses `<span>` with text content (`<span class="material-symbols-outlined">home</span>`), not `<i>` with class-based names. Detection patterns that only match `<i>` elements will miss Material Symbols entirely.
**Why it happens:** The other 3 libraries all use `<i>` elements. It is easy to assume all icon libraries do.
**How to avoid:** The aria-check hook must have a separate pattern for Material Symbols that matches `<span class="material-symbols-...">`.
**Warning signs:** Health and e-commerce projects (which use Material Symbols as primary) would have zero icon a11y enforcement.

### Pitfall 3: Reviewer Missing ICON-CATALOG.md
**What goes wrong:** The reviewer agent cannot perform icon consistency checks because ICON-CATALOG.md does not exist (system was generated before Phase 11, or user skipped /motif:system).
**Why it happens:** ICON-CATALOG.md is in `load_if_exists` for the reviewer, not `always_load`. This is correct -- the reviewer should still work without it -- but the icon checks need a graceful skip.
**How to avoid:** The reviewer's icon checks should be conditional: "IF ICON-CATALOG.md exists, perform icon consistency checks. IF NOT, skip and note: 'Icon catalog not found -- icon consistency checks skipped.'"
**Warning signs:** Reviewer crashes or produces empty icon sections when running against pre-Phase-11 projects.

### Pitfall 4: Decorative Icon aria-hidden on Lucide Elements
**What goes wrong:** After Lucide's `createIcons()` runs, the `<i data-lucide="...">` element is replaced with an inline `<svg>`. The hook checks the pre-rendered HTML (what the composer writes), but the rendered DOM is different.
**Why it happens:** Lucide is the only library that transforms its elements at runtime via JavaScript.
**How to avoid:** The hook checks the source code as written, not the rendered DOM. This is correct because the composer writes the `<i data-lucide>` elements. The hook should flag missing `aria-hidden` on the `<i>` element. When Lucide transforms it to `<svg>`, the `aria-hidden` attribute is preserved on the SVG.
**Warning signs:** None -- this is actually handled correctly by checking source code. Document it for clarity.

### Pitfall 5: Reviewer Cannot Grep for Icons Across Framework Syntaxes
**What goes wrong:** The reviewer tries to use literal grep patterns but JSX uses `className` instead of `class`, and template strings may split icon classes across lines.
**Why it happens:** JSX/TSX files use `className="ph ph-house"` not `class="ph ph-house"`. Vue files use `:class="..."` binding syntax. HTML files use `class="..."`.
**How to avoid:** The reviewer's grep patterns must account for both `class=` and `className=`. Provide both patterns in the reviewer instructions. For the hook, it already handles JSX via the existing `TARGET_EXTENSIONS` array (tsx, jsx, vue, html).
**Warning signs:** Fintech screens in React/Next.js projects would have zero icon findings because `className` does not match `class=`.

## Code Examples

### Example 1: Icon Element Detection in aria-check.js

```javascript
/**
 * Detect icon elements across all 4 curated libraries.
 * Returns array of { index, element, library } objects.
 */
function findIconElements(content) {
  const icons = [];
  const patterns = [
    { library: 'phosphor',  pattern: /<i\b[^>]*class="[^"]*\bph(?:-\w+)?\s+ph-[\w-]+[^"]*"[^>]*>[^<]*<\/i>/gi },
    { library: 'lucide',    pattern: /<i\b[^>]*data-lucide="[\w-]+"[^>]*>[^<]*<\/i>/gi },
    { library: 'material',  pattern: /<span\b[^>]*class="[^"]*material-symbols-[\w]+[^"]*"[^>]*>[\w_]+<\/span>/gi },
    { library: 'tabler',    pattern: /<i\b[^>]*class="[^"]*\bti\s+ti-[\w-]+[^"]*"[^>]*>[^<]*<\/i>/gi },
  ];

  for (const { library, pattern } of patterns) {
    let match;
    while ((match = pattern.exec(content)) !== null) {
      icons.push({ index: match.index, element: match[0], library });
    }
  }

  return icons;
}
```

### Example 2: Icon-Only Button Detection

```javascript
/**
 * Check if a button containing an icon element is icon-only.
 * Returns true if the button has no visible text besides the icon.
 */
function isIconOnlyButton(buttonElement, iconElement) {
  // Remove the icon element from the button's inner content
  const innerContent = buttonElement
    .replace(/^<button[^>]*>/, '')
    .replace(/<\/button>$/, '')
    .replace(iconElement, '')
    .trim();

  // Check if any visible text remains (ignore whitespace and HTML comments)
  const visibleText = innerContent
    .replace(/<!--.*?-->/g, '')
    .replace(/<[^>]*>/g, '')
    .trim();

  return visibleText.length === 0;
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| aria-label on all icon elements | Visually hidden text preferred over aria-label | WCAG 2.2 (2023+) | aria-label is not translated by browsers; visually hidden text IS translated. However, for Motif's CSS-class icon approach (not SVG), aria-label is acceptable since icons render as font glyphs, not text content. |
| SVG icons with role="img" | SVG with aria-hidden="true" when decorative | Ongoing best practice | Decorative SVGs should be hidden from assistive tech. Only informative standalone SVGs need role="img". |
| Generic "icon" a11y checks | Library-specific pattern detection | Motif Phase 12 | The 4 curated libraries have distinct HTML patterns. Generic detection produces false positives/negatives. |

**Deprecated/outdated:**
- `focusable="false"` on SVG: Primarily an IE11 concern. Still harmless to include but not strictly necessary for modern browsers. The Motif icon pipeline uses CSS class/font icons, not inline SVG (except Lucide post-render), so this is mostly irrelevant.

## Open Questions

1. **Should ICON-CATALOG.md move to always_load for the reviewer?**
   - What we know: It is currently in `load_if_exists`. The reviewer can function without it, but icon checks will be skipped.
   - What's unclear: Is it worth the context budget (1000 tokens) when icon checks are a secondary concern vs. the 4 primary lenses?
   - Recommendation: Keep as `load_if_exists`. The reviewer already has heavy always_load requirements (tokens.css, COMPONENT-SPECS.md, DESIGN-RESEARCH.md, screen source). Icon checks are additive, not blocking. The reviewer should gracefully skip if catalog is absent.

2. **Should the aria-check hook handle `<a>` elements the same as `<button>` elements?**
   - What we know: Icon-only links (`<a href="..."><i class="ph ph-link"></i></a>`) have the same a11y problem as icon-only buttons.
   - What's unclear: The current success criteria only mention "icon-only buttons." Links may be out of scope.
   - Recommendation: Include `<a>` elements in the icon-only check. The a11y concern is identical. The success criteria mention buttons specifically, but the fix is trivial to generalize and would be negligent to omit.

3. **How to handle the core/ vs .claude/get-motif/ sync for modified files?**
   - What we know: Phase 11 Plan 03 established the pattern of copying core/ source files to .claude/get-motif/ installed locations.
   - What's unclear: Whether the reviewer changes need to go through core/ first or can be written directly to .claude/get-motif/.
   - Recommendation: Follow Phase 11's established pattern. Modify the source files (if they exist in core/) then sync to .claude/get-motif/. For the hook (motif-aria-check.js), the installed copy IS the source -- hooks live only in .claude/get-motif/hooks/.

4. **What about the Lucide-specific `aria-hidden` scenario?**
   - What we know: Lucide replaces `<i data-lucide>` with `<svg>` at runtime. The composer writes the `<i>` element. The hook checks the `<i>` element.
   - What's unclear: Whether `aria-hidden="true"` on the `<i>` element transfers to the `<svg>` when Lucide transforms it.
   - Recommendation: LOW confidence on this transfer behavior. The hook should flag missing `aria-hidden` on the source `<i>` element regardless. If Lucide does not transfer the attribute, the developer will need to handle it. This is documented but not a blocking concern for Motif's pipeline -- Motif generates static HTML prototypes, not production runtime code.
