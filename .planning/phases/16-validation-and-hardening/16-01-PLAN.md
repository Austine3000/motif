  <action>
Create `scripts/compose-validator.js` following the exact pattern of `scripts/project-scanner.js` (shebang, 'use strict', JSDoc header, pure Node.js with fs and path only).

CLI interface:
```
node scripts/compose-validator.js --screen <name> --files <f1> <f2> ... [--scan-dir <dir>]
```
- `--screen`: screen name (for reporting)
- `--files`: list of generated file paths (space-separated, all args after --files until next flag or end)
- `--scan-dir`: directory containing PROJECT-SCAN.md and COMPONENT-GAP.md (default: `.planning/design`)

Parse CLI args manually (no yargs/commander) — same approach as project-scanner.js.

Implement three validation checks:

**1. Import Cycle Detection (ERROR — blocks commit)**
- For each file in --files, read its contents and extract import paths using regex: `/import\s+(?:\{[^}]+\}|\w+)\s+from\s+['"]([^'"]+)['"]/g`
- Only track imports that start with `./` or `../` or `@/` — skip bare module imports (react, next, etc.)
- Resolve relative imports to absolute paths. Filter to only imports that resolve to OTHER files in the --files list. External project files and node_modules are excluded from cycle detection.
- Build an adjacency list (Map<string, string[]>) of file-to-file dependencies
- Run DFS cycle detection with visited set and recursion stack (standard textbook algorithm from research)
- If cycles found: add to errors array with type "import-cycle" and detail showing the cycle path

**2. Naming Conflict Detection (ERROR — blocks commit, brownfield only)**
- Determine brownfield mode: check if `{scan-dir}/PROJECT-SCAN.md` exists. If not, skip this check entirely.
- For each file in --files, check `git status --porcelain` output (run once, parse output)
- If a file shows as `M` (modified) rather than `A` or `??` (new/untracked), that means the composition overwrote an existing project file — that is a naming conflict
- Skip files whose paths start with `.planning/` (greenfield output path — no conflicts possible)
- If conflicts found: add to errors array with type "naming-conflict"

**3. Missing Prop Detection (WARN — does not block, brownfield only)**
- If `{scan-dir}/COMPONENT-GAP.md` does not exist, skip entirely
- Read COMPONENT-GAP.md and parse the "Existing" and "Partial Match" tables to get component names and their file paths
- For each generated file, find JSX usage of existing components (regex: `<ComponentName` where ComponentName is in the existing/partial list)
- Extract props passed in JSX (regex for prop names before `=` or self-closing)
- If `{scan-dir}/PROJECT-SCAN.md` exists, check its Props Summary for declared props of that component
- If a prop is passed that is NOT in the declared props, add to warnings array with type "missing-prop"
- This is a best-effort check — regex-based, consistent with project-scanner.js approach

Output JSON to stdout:
```json
{
  "status": "pass|fail|warn",
  "screen": "{SCREEN_NAME}",
  "errors": [{ "type": "import-cycle|naming-conflict", "detail": "...", "severity": "error" }],
  "warnings": [{ "type": "missing-prop", "detail": "...", "severity": "warn" }]
}
```
- status = "fail" if errors.length > 0
- status = "warn" if errors.length === 0 and warnings.length > 0
- status = "pass" if both empty
- Exit code 0 for pass/warn, exit code 1 for fail

Use `child_process.execSync` for `git status --porcelain` (naming conflict check only). This is the same approach used by other scripts in the project.

Keep the script under 250 lines. Organize into clearly separated functions: parseArgs, extractLocalImports, detectCycles, checkNamingConflicts, checkMissingProps, main.
